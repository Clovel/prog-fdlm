# Spec 1 — Data layer (Drizzle + PostgreSQL + seeds + DB-backed read path)

**Status:** approved
**Date:** 2026-05-30
**Scope:** first of three specs. Spec 2 adds BetterAuth; Spec 3 adds the admin backoffice. This spec ships a working public site that reads its events from a Postgres database instead of TypeScript fixtures, with no auth and no write endpoints.

---

## 1. Goals

- Move event data from `src/fixtures/events-YYYY.tsx` into a Postgres database, accessed via Drizzle.
- Keep the existing fixtures on disk — they become the source for a `pnpm db:seed` script, not runtime imports.
- Introduce edition-scoped routing (`/[year]`) with `/` redirecting to the latest edition; old editions remain browsable.
- Expose four read-only REST endpoints that the public page calls client-side.
- Add a new entity, `general_alerts`, scoped per edition (banner-style notices on the public page). No public render in Spec 1 — the API serves them; the banner component lands when authoring exists (Spec 3).
- Ship behind no feature flag: when Spec 1 merges, the public site flips from fixture-driven to DB-driven.

## 2. Non-goals

- Auth, users, sessions, roles — Spec 2.
- Admin/backoffice UI, write endpoints, mutations, optimistic updates — Spec 3.
- Geocoding cache (lat/lng columns + write-back). The map keeps using runtime client-side `react-geocode`. Revisit when this becomes a perf problem.
- Server-side rendering of the public event list (RSC reading the DB directly). Spec 1 uses client fetch against API routes. Migration to RSC later is a local change because every endpoint is backed by a `getX()` query function.
- Full-text search (`pg_trgm`, `tsvector`). The `q` query param uses `ILIKE` only.
- Pretty event/edition slugs. Spec 1 uses UUIDs in event URLs.
- A client-side query cache (TanStack Query / SWR). HTTP `Cache-Control` headers do the heavy lifting; client cache lands in Spec 3 when mutations arrive.
- A public-page `<GeneralAlertsBanner>` component. API exposes them; UI ships with Spec 3.

## 3. Stack & file layout

- **DB:** PostgreSQL. Self-hosted (user-managed instance). App reaches it via `DATABASE_URL`.
- **Driver:** `postgres` (postgres.js).
- **ORM:** `drizzle-orm` with `drizzle-kit` for migrations and studio.
- **Validation:** existing `zod` (already in `package.json`).
- **No new framework deps** beyond Drizzle and the Postgres driver.

## 4. Database schema

### 4.1 Enums (`pgEnum`)

- `alert_variant` — `'default' | 'destructive' | 'warning' | 'success'`. Mirrors the variants supported by `src/components/ui/alert.tsx`. If new variants are added there, the enum and a follow-up migration must mirror them.
- `event_status` — `'canceled' | 'postponed' | 'rescheduled'`. Mirrors `Event['status']`.
- `embed_platform` — `'instagram' | 'facebook'`. Mirrors `EventEmbedLinkType`.
- `event_category` — exact values from `src/types/eventCategories.ts` (`'Centre ville'`, `'St. Michel'`, …, `'Saint-Médard-en-Jalles'`). The TypeScript const array remains the source of truth; the migration mirrors it.

### 4.3 Relations

```
editions ────┬─< events ──┬─< event_links
             │            ├─< event_embed_links
             │            └─< event_alerts
             └─< general_alerts
```

(`──<` = one-to-many.) In Drizzle `relations(...)`: `editions` hasMany `events` and `generalAlerts`; `events` belongsTo `edition`, hasMany `eventLinks`, `eventEmbedLinks`, `eventAlerts`; child tables belongsTo their parent.

### 4.4 Standalone-managed vs inline-managed entities

All six tables are entities in the data model; most have parent FKs with `ON DELETE CASCADE`. The distinction below is about **how the backoffice manages them** (Spec 3 UX), not about the data model:

- **Standalone-managed** — own list view and own edit page in the backoffice. Admin creates, edits, deletes them one by one.
  - `editions` — root entity.
  - `events` — listed and edited per edition (e.g. `/admin/editions/2026/events`).
  - `general_alerts` — although FK-scoped to editions, has its own CRUD UI (e.g. `/admin/editions/2026/alerts`). Listed, created, edited, deleted individually.
- **Inline-managed** — no standalone UI. Edited as repeatable form sections inside the parent's edit page. Cascade-delete with the parent.
  - `event_links`, `event_embed_links`, `event_alerts` — all rendered as repeatable rows on the event edit page.

Data model relationships (unchanged from §4.3):

```
editions ────┬─< events ──┬─< event_links
             │            ├─< event_embed_links
             │            └─< event_alerts
             └─< general_alerts
```

## 5. API surface

All four endpoints are GET, public, read-only, JSON-only. They live under `src/app/api/`. Validation uses Zod schemas defined alongside each route.

All responses set `Cache-Control: public, s-maxage=60, stale-while-revalidate=300`.

Errors:
- `400` for query/path validation failure, with `{ error: 'invalid_request', issues: [...] }`.
- `404` for unknown editions / events, with `{ error: 'not_found' }`.
- `500` for unexpected errors, with `{ error: 'internal_error' }`. Logged server-side.

### 5.1 `GET /api/editions`

Returns the list of editions, ordered `year DESC`. Used by the future edition switcher / footer past-editions link.

Response:

```ts
{
  editions: Array<{
    id: string;
    year: number;
    description: string | null;
  }>;
}
```

### 5.2 `GET /api/editions/[year]`

Returns edition metadata plus its `general_alerts` where `isPublished = true`. `general_alerts` is included here (not as its own endpoint) because the banners are small and always wanted alongside the metadata.

Path params: `year` — coerced to integer; 400 if not a valid integer.

Response:

```ts
{
  edition: {
    id: string;
    year: number;
    description: string | null;
    dayOfFestival: string;        // ISO date
  };
  generalAlerts: Array<{
    id: string;
    variant: AlertVariant;
    title: string | null;
    content: string;
    position: number;
  }>;
}
```

404 when the year is not in `editions`.

### 5.3 `GET /api/editions/[year]/events`

The main list endpoint. Returns event summaries.

Query params (all optional, all forward-compatible):

- `category` — must match `event_category` enum value; 400 otherwise.
- `q` — free-text. Server-side `ILIKE %q%` against `name`, `locationName`, and `array_to_string(artists, ' ')`. Trimmed; empty string ignored. Max length 200 chars.
- `genre` — single value; matches events whose `genres` array contains it (`?? '{val}'::text[]`).
- `status` — must match `event_status` enum; 400 otherwise.
- `ids` — comma-separated UUIDs (max 200). Filters to only those events. Invalid UUIDs → 400. Powers future favourites: client passes its locally-stored IDs, server returns just those (still scoped to the path's edition).
- `cursor` — opaque base64-encoded `(startTime, id)` tuple. Server decodes, filters `(startTime, id) > (cursor.startTime, cursor.id)`. Invalid cursors → 400.
- `limit` — default 100, min 1, max 200.

Sort order: `startTime ASC, id ASC` (deterministic; matches the cursor).

`EventSummary` shape:

```ts
{
  id: string;
  editionId: string;
  name: string | null;
  category: EventCategory | null;
  status: EventStatus | null;
  genres: string[] | null;        // included; cheap and used by category headers / future filters
  startTime: string;              // ISO
  endTime: string | null;
  priceText: string | null;
  location: {
    name: string;
    address: string | null;       // for client-side react-geocode
  };
  hasDescription: boolean;
  linkCount: number;
  embedCount: number;
  alertCount: number;
}
```

Response:

```ts
{
  events: EventSummary[];
  nextCursor: string | null;       // null when no more pages
}
```

`hasDescription`, `linkCount`, `embedCount`, `alertCount` are computed via subqueries / aggregates so the client knows whether expansion will produce content without paying for the heavy payload upfront.

### 5.4 `GET /api/events/[eventId]`

The detail endpoint. Called when a user expands an event card (`<Collapse>` open). Returns the heavy fields the summary omits.

Path params: `eventId` — must be a valid UUID; 400 otherwise. 404 if not found.

Response:

```ts
{
  event: {
    id: string;
    editionId: string;
    description: string | null;
    artists: string[] | null;
    links: Array<{ url: string; label: string }>;          // ordered by position
    embedLinks: Array<{ platform: 'instagram' | 'facebook'; url: string }>;   // ordered
    alerts: Array<{ variant: AlertVariant; title: string | null; content: string }>;   // ordered
  };
}
```

Implementation note: the four sibling queries (event row, links, embeds, alerts) are issued in parallel with `Promise.all`, not joined. With 1–10 children per event the join overhead isn't worth the SQL complexity.

## 6. Read path migration in the Next app

### 6.1 Routing

- `src/app/page.tsx` (root) — server component. Queries `SELECT year FROM editions ORDER BY year DESC LIMIT 1`. Calls `redirect('/' + year)`. If no editions exist yet (empty DB before first seed), returns a minimal "no edition yet" page.
- `src/app/[year]/page.tsx` — `'use client'`. The current `page.tsx` body moves here. Replaces the `import { events } from 'fixtures/events-2024'` with two parallel fetches against `/api/editions/[year]` and `/api/editions/[year]/events?limit=200`. Renders loading + error + empty states. The category grouping (`reduceEventsByCategory`) runs on the fetched array, unchanged.
- `src/app/[year]/not-found.tsx` — new file rendering a 404 message and a link back to `/`.
- The `[year]` segment validates the param — non-integer paths route to `not-found.tsx`. The page itself trips `notFound()` when the API returns 404.

### 6.2 Component changes

- `Header.tsx` — currently does `import { events } from 'fixtures/events-2024'` for `events.length`. Refactored to take a `count: number` prop from the page. The page passes `events.length` after the fetch resolves (or 0 during loading). Header text gets a graceful loading state ("Chargement…" or similar).
- `EventsMap.tsx` — already takes events as props. No code change beyond confirming it tolerates the now-async data (initial render with empty array, re-render when array arrives). Continues to runtime-geocode `addressStr` via `react-geocode` client-side.
- All other components (`EventCategoryView`, `EventList`, `EventListItem`, `EventListItemDetails`, `EventRender`, …) take props that match the existing `Event` shape. They keep working unchanged as long as the client builds an `Event`-shaped object from the API payload. A small `apiEventToView()` adapter in `src/app/[year]/page.tsx` hydrates the two API responses into the array shape `EventCategoryView` expects.

The original `src/types/Event.ts` stays as the view-layer shape. The DB-row types live alongside the schema (Drizzle's `InferSelectModel`). The API DTOs live next to each route. The adapter sits in the page.

### 6.3 Detail-on-expand

Spec 1 fetches the **summary** list up front and the **detail** only when a user expands a card. This is a behavioural change vs the current site, which has every event's description ready immediately because it's all bundled in the fixture import.

`EventListItem` already wraps its details in a `<Collapse>` triggered by a click. On first open per event, it fires `fetch('/api/events/' + id)`, shows a small skeleton in the details area, then renders. A tiny in-memory cache on the page component (a `Map<id, EventDetail>`) avoids re-fetching on subsequent opens of the same card within one session.

This in-memory cache is the minimum needed; TanStack Query / SWR remains deferred to Spec 3.

## 7. Seed strategy

**Goal:** `pnpm db:seed` on a fresh or partially-populated DB ends up with both 2023 and 2024 fully populated, idempotently. Re-runs reflect the current fixture state.

### 7.1 Behaviour

`src/db/seed/index.ts`:

1. Connects via the same `src/db` Drizzle client.
2. **Editions upsert.** Hardcoded list `[{ year: 2023, dayOfFestival: '2023-06-21' }, { year: 2024, dayOfFestival: '2024-06-21' }]`. For each: `INSERT ... ON CONFLICT (year) DO UPDATE SET description = EXCLUDED.description, dayOfFestival = EXCLUDED.dayOfFestival, updatedAt = NOW()`.
3. **For each fixture file, for each event:**
    - Map fixture shape → DB row shape (see §7.2).
    - `INSERT ... ON CONFLICT (editionId, legacyId) DO UPDATE SET <editable cols> = EXCLUDED.<col>, updatedAt = NOW() RETURNING id`. Returns the event's UUID either way.
    - **Inside a single transaction per event, for each child table (`event_links`, `event_embed_links`, `event_alerts`):** Strategy B — position-keyed upsert plus trailing delete.
      1. For each child in the fixture array (at index `i`):
          ```sql
          INSERT INTO event_links (eventId, url, label, position)
          VALUES ($eventId, $url, $label, $i)
          ON CONFLICT (eventId, position) DO UPDATE
            SET url = EXCLUDED.url, label = EXCLUDED.label;
          ```
          (For `event_embed_links` update `platform, url`; for `event_alerts` update `variant, title, content`.)
      2. Delete trailing rows where the fixture array is now shorter:
          ```sql
          DELETE FROM event_links WHERE eventId = $eventId AND position >= $fixtureLength;
          ```
4. Same strategy for `general_alerts` keyed on `(editionId, position)`. (Note: in Spec 1 the fixtures have no `generalAlerts`, so this is dead code for now — kept to keep the seed shape uniform when general alerts get seeded later.)
5. Logs per-edition: `N events upserted, M child rows upserted, K trailing rows deleted`.

**Why Strategy B (position-keyed upsert) rather than delete-then-insert:** preserves child-row UUIDs across re-seeds — useful so that backoffice-bookmarked links to specific children survive a re-seed. Reflects fixture deletions (trailing-delete step). The trade-off is that reordering children in the backoffice (Spec 3) requires the standard position-swap pattern to avoid colliding with the UNIQUE constraint (temporarily set one row to a `-1` position, swap the others, then move the parked row back). This is a known pattern and not unique to this app.

**Documented warning** in the seed file header: *"This script is a bootstrap. In production it is run once after the first deploy. Re-running it overwrites any backoffice edits to seeded events (the event row content) but preserves child row identities. Use `db:reset` (drop + migrate + seed) only in development."*

### 7.2 Fixture → row mapping

- `id: string` → `events.legacyId`. New `events.id` is `gen_random_uuid()`.
- `name?`, `description?`, `category?`, `status?`, `genres?`, `artists?` → matching columns, nullable.
- `description: React.ReactNode` → asserted to be a string (descriptions were migrated to markdown in commit 7bc165b). Throws on JSX.
- `price?: number | string` → `priceText: string | null`. Numbers `String()`-cast (defensive — fixtures only use strings).
- `location.name` → `locationName`.
- `location.addressStr?` → `locationAddress`.
- `location.coords?` → ignored in Spec 1 (no coord columns). Logged as a warning if present so we don't silently drop information.
- `startTime: Date`, `endTime?: Date` → stored as `timestamptz`. Fixture dates use bare `new Date('2024-06-21T19:00:00')` which JS parses in the runtime's local timezone — risky for cross-machine reproducibility. `src/db/seed/normalizeTime.ts` parses each bare ISO string explicitly as Europe/Paris using `date-fns-tz`. Strings with explicit offsets (e.g. `2023-06-21T18:00:00+02:00`) parse without help. The helper accepts either `Date` or `string`; if the fixture passes a `Date` already constructed in unknown TZ, we serialize it as Europe/Paris first.
- `links[].label: React.ReactNode` → `event_links.label: text`. The seed flattens any `React.ReactNode` (including JSX subtrees) to plain text via a small helper before inserting; the 2024 fixture contains at least one JSX label.
- `links[]` → `event_links`, ordered by array index.
- `embedLinks[]` → `event_embed_links` with `platform = type` and array-index `position`.
- `alerts[]` → `event_alerts`. `type` mapped to `variant`; must match `alert_variant` enum, throws otherwise.

### 7.3 `db:reset` (dev-only convenience)

A `pnpm db:reset` script (documented in README, not added to CI) does `drizzle-kit drop && drizzle-kit push && pnpm db:seed`. Convenient for local iteration; explicitly *not* for production.

### 7.4 Production seeding

After the first deploy with the new schema:

1. Run migrations against production: `pnpm db:migrate`.
2. Run the seed once: `pnpm db:seed`.

The seed is **not** wired into the deploy step. Future deploys run `db:migrate` only.

## 8. Rollout plan

1. **Add dependencies.** `drizzle-orm`, `postgres`, `date-fns-tz`. Dev: `drizzle-kit`, `tsx`, `@types/pg` (if needed by helpers — postgres.js doesn't need it).
2. **Config files.** `drizzle.config.ts` at repo root pointing at `src/db/schema` and `src/db/migrations`. `.env.example` documenting `DATABASE_URL`.
3. **Schema files** under `src/db/schema/`. Six tables + `enums.ts` + `relations.ts` + `index.ts`.
4. **Initial migration.** `pnpm db:generate` produces `src/db/migrations/0000_*.sql`. Commit the SQL. If the target Postgres is <13, prefix with `CREATE EXTENSION IF NOT EXISTS pgcrypto;` so `gen_random_uuid()` resolves (Postgres 13+ has it built-in and the extension call is a no-op).
5. **DB client.** `src/db/index.ts` exports a HMR-safe cached client.
6. **Query functions** under `src/db/queries/`: `listEditions`, `getEdition`, `listEditionEvents`, `getEventDetail`. Each pure, takes the Drizzle client + params, returns shaped DTOs.
7. **Seed script** under `src/db/seed/`. Add `db:*` scripts to `package.json`.
8. **API route handlers** under `src/app/api/`. Each handler parses with Zod, calls the matching query function, sets cache headers, returns JSON.
9. **Root server component.** Rewrite `src/app/page.tsx` to query the latest year and redirect.
10. **`/[year]` page.** Move current `page.tsx` body into `src/app/[year]/page.tsx`, swap fixture import for two parallel `fetch()`s, render loading/error/empty states. Wire the `apiEventToView()` adapter.
11. **`/[year]/not-found.tsx`.** Minimal 404.
12. **Header refactor.** Take `count` as a prop. Remove fixture import.
13. **Detail-on-expand wiring.** Update `EventListItem` to fetch `/api/events/[id]` on first `<Collapse>` open and pass into `EventListItemDetails`. In-memory `Map` cache at the page level.
14. **EventsMap sanity check.** Confirm it tolerates the async / initially-empty array.
15. **Delete fixture imports from runtime.** Verify `grep -r "from 'fixtures/" src/app src/components` returns nothing.
16. **Local verification.**
    - Create local Postgres DB (Docker or system).
    - `pnpm db:migrate && pnpm db:seed && pnpm dev`.

## 9. Risk register

- **Cold-start RTT.** The old fixture import was 0 RTT; the new flow is two parallel fetches + a detail fetch per expand. Mitigated by `Cache-Control: s-maxage=60, stale-while-revalidate=300` (edge caching) and `Promise.all` parallel fetches. Worst case on cold edge: ~200–400 ms slower first paint. Acceptable.
- **Postgres availability.** If the DB is unreachable, the public page shows an error state. Mitigated by SWR-on-edge: last good response served for 5 min while origin is down. The `react-geocode` map continues to work because the map only depends on event addresses already in the cached payload.
- **Timezone drift in seeded times.** Bare ISO strings in fixtures are interpreted in the runtime's local TZ, which varies across dev machines. Mitigated by `normalizeTime.ts` parsing them explicitly as Europe/Paris. Post-seed smoke check: pick three known events and assert their stored UTC values round-trip to the fixture's wall-clock time in `Europe/Paris`.
- **Loss of immediate descriptions.** Old fixture-driven site had every description ready instantly; new flow defers description load until expand. Mitigated by tiny payload per detail and skeleton state in the expanded card. UX trade-off accepted: first paint is faster, expansion is slightly slower (a few hundred ms on a cold edge, often instant on warm).
- **Seed overwrites admin edits to seeded events.** Spec 3 risk, not Spec 1: once authoring exists, re-running the seed against an already-populated DB will overwrite event-row content and any link/embed/alert content at matching positions. Child-row identities (UUIDs) are preserved by Strategy B. Documented in the seed file header. Production seed is a one-shot.

## 10. Decisions captured

| Question                                  | Decision                                                                 |
| ----------------------------------------- | ------------------------------------------------------------------------ |
| DB engine                                 | PostgreSQL (self-hosted)                                                 |
| ORM                                       | Drizzle (`pg` dialect via `postgres` driver)                             |
| Edition selection on public page          | URL slug `/[year]`; `/` redirects to latest edition                      |
| Geocoding                                 | No caching in Spec 1. Keep current client-side runtime `react-geocode`.  |
| Render model                              | Client fetch against API routes. RSC migration deferred.                 |
| General alerts scope                      | Edition-scoped; `isPublished` toggle, no time window.                    |
| Primary keys                              | UUIDs everywhere except `editions.year` UNIQUE; `editions.id` is UUID PK |
| Seed strategy                             | Upsert editions + events; position-keyed upsert + trailing delete for children (Strategy B) |
| Client query cache                        | None in Spec 1. Defer to Spec 3 when mutations arrive.                   |
| Public render of general alerts           | API serves them; UI banner ships in Spec 3                               |
